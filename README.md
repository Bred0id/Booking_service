# Booking Service

Backend-сервис для бронирования музыкальных студий.

Основная идея проекта — создать сервис, который позволяет пользователям просматривать студии, проверять расписание и создавать бронирования, а персоналу — отслеживать состояние помещений, оборудования и текущих бронирований.

Кроме базовых сценариев бронирования, проект предусматривает аналитические возможности: просмотр загрузки студий, активных бронирований, неоплаченных бронирований, проблемного оборудования и других показателей, полезных для управления сетью студий.

## Что реализовано

- просмотр списка музыкальных студий;
- получение подробной информации о студии;
- почасовое расписание студии на выбранный день;
- создание бронирования с проверкой доступности слота;
- обработка конфликта бронирований;
- просмотр отзывов по студиям;
- админский просмотр активных бронирований;
- SQL-представления для аналитики и служебных сценариев;
- запуск PostgreSQL и FastAPI через Docker Compose.

## Техническая идея

Проект построен вокруг связки FastAPI и PostgreSQL.

Часть бизнес-логики вынесена на уровень базы данных: создание бронирования, проверка пересечений по времени, расчёт стоимости, представления для чтения данных и триггеры для поддержания целостности.

API не дублирует эту логику в Python, а вызывает готовые функции и читает подготовленные представления из PostgreSQL.

## Предметная область

Сервис моделирует работу сети музыкальных студий.

В базе данных есть сущности:

- пользователи;
- студии;
- оборудование;
- бронирования;
- платежи;
- обслуживание оборудования;
- отзывы.

Основной пользовательский сценарий:

1. посмотреть список студий;
2. посмотреть информацию о конкретной студии;
3. посмотреть почасовое расписание студии;
4. создать бронирование;
5. получить ошибку, если выбранный слот уже занят.

## Особенности реализации

### Бизнес-логика на уровне PostgreSQL

Создание бронирования выполняется через функцию PostgreSQL:

```sql
music_studio.create_booking(...)
```

Функция:

- проверяет существование пользователя;
- проверяет статус пользователя;
- проверяет доступность студии;
- рассчитывает стоимость бронирования;
- создаёт запись;
- возвращает `booking_id`.

Если выбранное время пересекается с уже существующим бронированием, база возвращает ошибку, а API преобразует её в HTTP-ответ `409 Conflict`.

### Почасовое расписание студии

Для студии можно получить расписание на выбранный день:

```http
GET /studios/{studio_id}/schedule?day=2026-05-01
```

Расписание строится на стороне базы данных и показывает свободные и занятые часовые слоты.

### Разделение публичных и служебных ручек

Некоторые данные не должны быть общедоступными.

Например, список активных бронирований доступен только через админский endpoint:

```http
GET /admin/bookings/active
```

Для доступа нужно передать ключ в заголовке:

```http
key: dev-admin-key
```

В дальнейшем планируется расширить эту механику для аналитических запросов и отдельных уровней доступа.

## SQL-слой

SQL-скрипты выполняются при первом запуске контейнера PostgreSQL.

Порядок файлов:

1. `01_create.sql` — создание схемы и таблиц;
2. `02_fill.sql` — заполнение тестовыми данными;
3. `03_indexes.sql` — индексы;
4. `04_views.sql` — представления;
5. `05_functions.sql` — функции;
6. `06_triggers.sql` — триггеры.

В проекте используются:

- ограничения целостности;
- внешние ключи;
- индексы;
- представления;
- PL/pgSQL-функции;
- триггеры.

## Переменные окружения

Создайте файл `.env` в корне проекта.

Пример:

```env
POSTGRES_DB=Project
POSTGRES_USER=postgres
POSTGRES_PASSWORD=postgres

DATABASE_URL=postgresql://postgres:postgres@db:5432/Project

ADMIN_KEY=dev-admin-key
ANALYTICS_KEY=dev-analytics-key
```

Файл `.env` не должен попадать в репозиторий.

Для примера можно использовать `.env.example`:

```env
POSTGRES_DB=Project
POSTGRES_USER=postgres
POSTGRES_PASSWORD=postgres

DATABASE_URL=postgresql://postgres:postgres@db:5432/Project

ADMIN_KEY=your-admin-key
ANALYTICS_KEY=your-analytics-key
```

## Запуск

Из корня проекта:

```bash
docker compose up --build
```

Если нужно пересоздать базу данных и заново применить SQL-скрипты:

```bash
docker compose down -v
docker compose up --build
```

После запуска API доступен по адресу:

```text
http://127.0.0.1:8000
```

Документация Swagger:

```text
http://127.0.0.1:8000/docs
```

## Примеры запросов

### Проверка работы сервиса

```bash
curl http://127.0.0.1:8000/
```

Ответ:

```json
{
  "status": "ok"
}
```

### Получить список студий

```bash
curl http://127.0.0.1:8000/studios
```

### Получить информацию о студии

```bash
curl http://127.0.0.1:8000/studios/1
```

Если студия не найдена:

```json
{
  "detail": "Studio not found"
}
```

### Получить расписание студии на день

```bash
curl "http://127.0.0.1:8000/studios/1/schedule?day=2026-05-01"
```

Пример ответа:

```json
[
  {
    "studio_id": 1,
    "studio_name": "Класс камерного ансамбля",
    "slot_start": "2026-05-01T10:00:00",
    "slot_end": "2026-05-01T11:00:00",
    "slot_status": "free"
  }
]
```

### Создать бронирование

```bash
curl -i -X POST http://127.0.0.1:8000/bookings \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": 3,
    "studio_id": 2,
    "start": "2026-05-01T10:00:00",
    "end": "2026-05-01T12:00:00",
    "purpose": "Тестовое бронирование"
  }'
```

Успешный ответ:

```json
{
  "booking_id": 181
}
```

Если слот уже занят:

```json
{
  "detail": "Booking time overlaps with another booking"
}
```

HTTP-статус:

```text
409 Conflict
```

### Получить активные бронирования

Админский endpoint:

```bash
curl -H "key: dev-admin-key" http://127.0.0.1:8000/admin/bookings/active
```

Без ключа:

```json
{
  "detail": "Access denied"
}
```

### Получить отзывы по студии

```bash
curl http://127.0.0.1:8000/reviews/1
```
